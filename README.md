# Журнал работ на строительном объекте (Construction Work Log System)

Production-ready веб-приложение для ведения и контроля журнала выполненных строительных работ на объекте. Позволяет фиксировать объемы работ, единицы измерения, даты выполнения и ФИО исполнителей с поддержкой справочника видов работ.

Проект разработан в рамках демонстрации коммерческих подходов к разработке на стеке **NestJS, Prisma, PostgreSQL, React (Vite) и Material UI (DataGrid)**.

---

## 🚀 Быстрый запуск (Docker Compose)

Вся система упакована в Docker-контейнеры и запускается одной командой. Перед запуском убедитесь, что порты `3000`, `3001` и `5432` свободны.

1. **Клонируйте проект и перейдите в корень:**
   ```bash 
   construction-work-log
   ```

2. **Запустите оркестрацию контейнеров:**
   ```bash
   docker compose up --build
   ```

3. **Что произойдет автоматически:**
   - Поднимется база данных **PostgreSQL** и будет проверен ее статус готовности (healthcheck).
   - Скомпилируется и запустится **NestJS API** (на порту `3001`). Он автоматически применит миграции базы данных и запустит скрипт **seed**, наполняющий базу базовыми справочными данными (виды работ и несколько тестовых записей журнала).
   - Скомпилируется и запустится **Vite React Frontend** (на порту `3000`) под управлением легковесного веб-сервера **Nginx**, настроенного для работы с Single Page Applications (SPA).

4. **Откройте приложение в браузере:**
   - Клиентский интерфейс: [http://localhost:3000](http://localhost:3000)
   - Swagger / REST API: [http://localhost:3001/api](http://localhost:3001/api)

## 🛠️ Запуск в режиме разработки (Локально)

Если вы хотите вносить изменения в код с поддержкой горячей перезагрузки (hot reload) без контейнеров:

### Шаг 1: База данных
Запустите только PostgreSQL через docker-compose:
```bash
docker compose up -d postgres
```

### Шаг 2: Настройка бэкенда (`backend/`)
1. Установите зависимости и сгенерируйте клиент Prisma:
   ```bash
   cd backend
   npm install
   npx prisma generate
   ```
2. Скопируйте `.env` файл из корня в `backend/.env` или используйте переменные по умолчанию.
3. Выполните миграции и наполните БД данными:
   ```bash
   npx prisma migrate dev --name init
   npm run prisma:seed
   ```
4. Запустите сервер разработки:
   ```bash
   npm run start:dev
   ```
   API будет доступно по адресу [http://localhost:3001/api](http://localhost:3001/api).

### Шаг 3: Настройка фронтенда (`frontend/`)
1. Установите зависимости:
   ```bash
   cd ../frontend
   npm install
   ```
2. Запустите Vite dev-сервер:
   ```bash
   npm run dev
   ```
   Интерфейс откроется на [http://localhost:3000](http://localhost:3000).

## 🏗️ Архитектура и Структура проекта

Код структурирован строго по промышленным стандартам с четким разделением ответственности.

### Бэкенд (NestJS)
Построен по модульному принципу:
- `src/database/` — управление подключением к PostgreSQL через Prisma ORM.
- `src/common/` — общие утилиты, включая глобальный `HttpExceptionFilter` (санитизирует ошибки базы данных и возвращает клиенту JSON единого формата).
- `src/modules/work-types/` — справочник видов работ.
- `src/modules/work-logs/` — журнал выполненных работ с DTO-валидаторами и фильтрацией.

### Фронтенд (React)
Реализован на базе методологии, близкой к **Feature-Sliced Design (FSD)**:
- `app/` — инициализация приложения, глобальные стили, роутинг и провайдеры (MUI, TanStack Query).
- `pages/` — страницы-компоненты (`WorkLogsPage`, `WorkTypesPage`), собирающие виджеты и управляющие локальным состоянием.
- `widgets/` — крупные самостоятельные блоки интерфейса (например, `WorkLogsTable` с MUI DataGrid).
- `features/` — интерактивные пользовательские действия с формами и валидацией (`WorkLogFormModal`, `WorkTypeFormModal`, `WorkLogFilters`).
- `entities/` — бизнес-сущности (`work-log`, `work-type`), их интерфейсы и хуки синхронизации с сервером на базе **TanStack Query**.
- `shared/` — переиспользуемые инфраструктурные элементы: API клиент (Axios), кастомные модалки (`ConfirmDialog`), провайдер уведомлений (`ToastProvider`), ErrorBoundary и утилиты.

## 📊 Проектирование Базы Данных (Prisma Schema)

Схема базы данных находится в нормализованной 3NF форме.

### 1. `work_types` (Справочник видов работ)
* `id` (UUID, Primary Key)
* `name` (VARCHAR(100), Unique) — Наименование вида работ (например, "Монтаж опалубки").
* `created_at` (Timestamp)

### 2. `work_logs` (Журнал работ)
* `id` (UUID, Primary Key)
* `date` (Timestamp) — Дата выполнения работ. На данное поле наложен **индекс** для оптимизации выборок с фильтрацией и сортировкой по дате.
* `volume` (Double Precision) — Объем выполненных работ (строго > 0).
* `unit` (VARCHAR(10)) — Единица измерения (м², м³, т, шт, м.п.).
* `worker_name` (VARCHAR(150)) — ФИО ответственного бригадира/исполнителя.
* `work_type_id` (UUID, Foreign Key) — Ссылка на справочник.
* `created_at` / `updated_at` (Timestamp)

*Связь:* `WorkType` **1 -> N** `WorkLog` с ограничением `onDelete: Restrict` (нельзя удалить вид работ из справочника, если в журнале есть записи по нему).

## 🔒 Валидация и Безопасность

- **Валидация на бэкенде:**
  Все входящие REST payloads валидируются через NestJS `ValidationPipe` на базе `class-validator`. Поля очищаются от нежелательных инъекций (`whitelist: true`), типы приводятся автоматически (`transform: true`).
- **Валидация на фронтенде:**
  Формы построены на связке `react-hook-form` и `zod`. Схемы валидации проверяют обязательность полей, длину строк, положительные числа для объемов и корректность ISO дат.
- **Обработка ошибок:**
  Любые сетевые сбои или ошибки валидации перехватываются глобальным `ErrorBoundary` на клиенте и `HttpExceptionFilter` на сервере. Ошибки отображаются пользователю в виде понятных тостов или подсвечивают невалидные поля формы.

## 🎨 Дизайн и UX

Интерфейс выглядит как строгий, стильный и современный внутренний инструмент управления строительными проектами:
- **Цветовая палитра:** Глубокий корпоративный бирюзовый (Teal `#006666`) в сочетании с профессиональным шиферным (Slate `#475569`).
- **DataGrid:** Премиальная таблица MUI DataGrid с поддержкой адаптивной верстки, кастомной сортировки, пагинации и интерактивных кнопок действий.
- **Empty & Loading States:** Таблица снабжена анимированными плейсхолдерами загрузки и красивыми пустыми состояниями с иконками, если фильтры не дали результатов.
- **Confirmations:** Опасные действия (удаление записей) запрашивают подтверждение в стильном диалоговом окне.
- **Notifications:** Любое действие (успешное добавление, изменение, удаление или ошибка) сопровождается анимированными всплывающими плашками в правом нижнем углу.
